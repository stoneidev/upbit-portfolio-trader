1. Pattern Context: Agentic Systems의 위치 선정

1.1 Workflow vs Agent: 근본적 구분

Anthropic은 "agentic system"을 두 범주로 구분합니다. 이 구분은 구현 아키텍처 선택의 첫 번째 분기점입니다.

구분 Workflow Agent
정의 LLM과 도구가 미리 정의된 코드 경로로 오케스트레이션 LLM이 자체적으로 프로세스를 동적 제어
제어 흐름 Deterministic (코드가 결정) Non-deterministic (모델이 결정)
예측 가능성 높음 낮음
적합 시나리오 반복적, 구조화된 작업 개방형, 탐색적 작업
AWS 매핑 Step Functions + Bedrock Bedrock Agents / Custom Agent Loop
Architecture Decision

Orchestrator-Worker는 이 스펙트럼의 중간에 위치합니다. Orchestrator 자체는 Agent적(동적 판단)이지만, Worker들은 Workflow적(정해진 역할 수행)입니다. 이 하이브리드 특성이 실용성의 핵심입니다.

1.3 Orchestrator-Worker vs Parallelization: 핵심 차이

이 두 패턴은 겉보기에 유사하지만, 아키텍처적으로 근본적인 차이가 있습니다:

관점 Parallelization Orchestrator-Worker
Task Decomposition Compile-time (코드에 하드코딩) Runtime (LLM이 입력 분석 후 결정)
Worker 수 고정 (N개 병렬) 가변 (입력에 따라 1~N)
Worker 선택 미리 결정됨 Orchestrator가 동적 선택
결과 통합 프로그래밍 로직 (concat, vote 등) LLM이 synthesis
비용 예측 Deterministic (고정 N회 호출) Non-deterministic (가변 호출)
AWS 구현 Step Functions Parallel State Step Functions + Dynamic Map State + Bedrock
Implementation Implication

Parallelization은 비용과 latency가 예측 가능하므로 SLA를 걸기 쉽습니다. Orchestrator-Worker는 동적이므로 비용 상한(budget cap)과 timeout 전략이 필수입니다. 이는 곧 maxConcurrency, maxIterations, budgetTokens 파라미터 설계로 이어집니다.

2. 이론적 기반: 4대 핵심 자료 분석

2.1 Anthropic — "Building Effective Agents" (2024.12)

Practitioner Guide 저자: Erik S., Barry Zhang

Augmented LLM: 모든 에이전트 패턴의 기본 단위

Anthropic은 에이전트를 구성하는 최소 단위를 "Augmented LLM"으로 정의합니다. 이는 bare LLM에 세 가지 capability를 결합한 것입니다:

구현 관점에서 이는 곧 각 Worker가 독립적인 Augmented LLM 인스턴스임을 의미합니다. Worker마다 다른 retrieval source, 다른 tool set, 다른 memory scope를 가질 수 있습니다.

Orchestrator-Worker: Anthropic의 정의

"A central LLM dynamically breaks down tasks, delegates them to worker LLMs, and synthesizes their results. The key difference from parallelization: subtasks aren't pre-defined but determined by the orchestrator based on the specific input."

Anthropic이 제시하는 적용 시나리오

  • Coding products: 여러 파일에 걸친 복잡한 변경. 어떤 파일을 수정해야 하는지, 몇 개나 수정해야 하는지가 입력에 따라 달라짐
  • Search tasks: 다수 소스에서 정보를 수집하고 분석. 어떤 소스가 관련 있는지 미리 알 수 없음

Anthropic의 설계 철학 (구현에 직접 영향)

  1. Start simple: 단일 LLM + 좋은 프롬프트로 시작. 복잡성은 단순한 방법이 실패할 때만 추가
  2. Transparency: Orchestrator의 계획 단계를 가시화. 디버깅과 신뢰 확보에 필수
  3. ACI (Agent-Computer Interface): 도구 설계에 HCI 수준의 투자 필요. 도구 문서화, 에지 케이스 처리, 형식 최적화가 프롬프트 튜닝보다 더 중요할 수 있음
Implementation Takeaway

Anthropic의 가장 중요한 메시지는 "프레임워크보다 기본기"입니다. Worker의 tool definition 품질이 전체 시스템 성능의 병목이 됩니다. 구현 시 tool schema 설계에 전체 effort의 30% 이상을 투자해야 합니다.

2.2 Andrew Ng — "Multi-Agent Collaboration" (DeepLearning.AI)

Conceptual Framework 저자: Andrew Ng

핵심 논제: 왜 Multi-Agent가 Single-Agent보다 나은가?

Ng은 세 가지 이론적 근거를 제시합니다:

1. 실증적 증거

Ablation study 결과, 에이전트 수를 줄이면 성능이 하락합니다. 이는 multi-agent 구조가 단순한 오버헤드가 아니라 실질적 성능 기여를 한다는 것을 의미합니다.

2. 인지적 집중 효과

LLM은 대용량 컨텍스트를 처리할 수 있지만, "한 번에 하나에 집중"하도록 프롬프트하면 성능이 향상됩니다. 이는 attention mechanism의 특성과 일치합니다.

3. 소프트웨어 공학적 분해

프로세스/스레드 분해와 동일한 원리. 각 에이전트를 독립적으로 테스트, 최적화, 교체할 수 있으므로 시스템 유지보수성이 향상됩니다.

역할 할당 메커니즘

동일한 LLM을 여러 번 호출하되, System Prompt를 통해 각기 다른 역할을 부여합니다. 이것이 "virtual team" 구성의 기술적 본질입니다: 

// Worker A: Software Engineer 역할
{
  "system": "You are a senior software engineer. Your sole responsibility is writing
             clean, efficient, production-ready code. You do NOT design systems or
             write tests. You implement exactly what the specification describes.",
  "model": "claude-sonnet-4-6-20250514"
}

// Worker B: QA Engineer 역할
{
  "system": "You are a QA engineer. Your sole responsibility is writing comprehensive
             test cases. Given code and specifications, identify edge cases, failure
             modes, and write tests that verify correctness.",
  "model": "claude-sonnet-4-6-20250514"
}

// Orchestrator: Tech Lead 역할
{
  "system": "You are a tech lead. Analyze the incoming request, decompose it into
             discrete implementation tasks, assign each to the appropriate specialist,
             and synthesize their outputs into a coherent deliverable.",
  "model": "claude-opus-4-6-20250514"  // Orchestrator는 더 강력한 모델 사용
}

Ng이 지적하는 한계

Predictability Risk

"다중 에이전트 협업의 출력 품질은 예측하기 어렵습니다. 특히 에이전트 간 자유 상호작용을 허용하고 여러 도구를 제공할 때 더욱 그렇습니다."

구현 시사점: 자유 상호작용(Agent-to-Agent 직접 통신)보다는 Orchestrator를 통한 중앙 집중식 통신이 예측 가능성을 높입니다. 이는 곧 Hub-and-Spoke 토폴로지 선택의 근거가 됩니다.

2.3 LangChain — "LangGraph Multi-Agent Workflows"

Implementation Framework 저자: LangChain Team

State Machine으로서의 Multi-Agent System

LangGraph의 핵심 추상화: 다중 에이전트 시스템은 방향 그래프(labeled directed graph)로 표현되는 상태 기계입니다.

  • Node = Agent 또는 Function (computation unit)
  • Edge = Transition rule (조건부 또는 무조건)
  • State = Graph를 통해 흐르는 공유 데이터

3가지 Orchestration Topology

구현 관점에서의 핵심 인사이트

Supervisor-as-Tool-User Pattern

"감독자는 다른 에이전트를 도구로 사용하는 에이전트다" — 이것이 LangGraph 구현의 핵심입니다. 기술적으로, Worker Agent를 Tool Definition으로 감싸면 Orchestrator의 기존 tool-use 메커니즘을 그대로 활용할 수 있습니다. 새로운 routing 로직이 필요 없습니다.

2.4 MetaGPT — "Meta Programming for Multi-Agent Collaborative Framework"

Empirical Research arXiv:2308.00352 | Sirui Hong et al. (Jürgen Schmidhuber 참여)

해결하는 문제: Cascading Hallucination

LLM을 단순히 체이닝하면 환각(hallucination)이 단계를 거칠수록 증폭됩니다. A의 작은 오류가 B에서 확대되고, C에서는 완전히 이탈합니다. 이는 naive한 multi-agent 구현의 가장 큰 실패 모드입니다.

해법: Standard Operating Procedures (SOP) as Prompt Sequences

실제 소프트웨어 회사의 개발 프로세스를 그대로 인코딩합니다. 자연어 대화가 아닌 구조화된 artifact를 통해 에이전트 간 통신합니다.

Communication Protocol: Shared Message Pool + Pub/Sub

MetaGPT의 핵심 아키텍처 혁신:

기존 방식 (ChatDev 등)

Agent-to-Agent 직접 대화. 자연어 기반.

문제: 대화가 길어지면 정보 손실, 맥락 이탈, hallucination 전파

MetaGPT 방식

공유 메시지 풀에 구조화된 문서를 게시. 구독 기반 소비.

장점: 정보 무손실, 역할 기반 필터링, 의존성 기반 활성화

// MetaGPT Communication Protocol (conceptual)
class SharedMessagePool {
  publish(role: Role, artifact: StructuredDocument): void;
  subscribe(role: Role, dependsOn: Role[]): Observable<StructuredDocument>;
}

// Engineer는 Architect의 설계서가 게시된 후에만 활성화
engineer.subscribe(["architect"]).pipe(
  filter(msg => msg.type === "SystemDesign"),
  map(design => engineer.implement(design))
);

Executable Feedback Mechanism

Engineer가 생성한 코드를 실제로 실행하여 검증합니다. 실패 시 PRD, 설계 문서, 기존 코드를 참조하여 디버깅합니다 (최대 3회 재시도). 이것이 "verification gate" 역할을 하며 hallucination 전파를 차단합니다.

Key Insight for Implementation

MetaGPT가 증명한 것: 구조화된 intermediate artifact + 실행 기반 검증이 자연어 체이닝 대비 극적으로 우수합니다. 구현 시 Worker 간 통신을 자연어가 아닌 schema-validated JSON/structured format으로 강제하고, 가능한 모든 단계에서 자동 검증 게이트를 삽입해야 합니다.

Cost-Quality Trade-off Analysis

MetaGPT는 ChatDev 대비 62% 더 많은 토큰을 소비합니다 (31,255 vs 19,292). 그러나 Human Revision Cost가 2.5회 → 0.83회로 67% 감소합니다.

ROI 계산: Senior Engineer의 시간을 $150/hr로 가정하고, 한 번의 revision에 30분이 소요된다면:

  • ChatDev: 2.5 × $75 = $187.50 인건비
  • MetaGPT: 0.83 × $75 = $62.25 인건비
  • 절감: $125.25/task (추가 토큰 비용 ~$0.36 대비)

추가 토큰 비용 대비 인건비 절감의 비율이 약 350:1입니다. 이것이 Orchestrator-Worker 패턴의 경제적 정당성입니다.

4.2 핵심 컴포넌트 상세 설계

A. Orchestrator — Planning & Decomposition Engine

interface OrchestratorConfig {
  model: string;                    // 가장 강력한 모델 사용 (claude-opus-4-6 권장)
  maxWorkers: number;               // 동시 실행 가능한 Worker 수 상한
  maxIterations: number;            // 전체 orchestration loop 상한
  budgetTokens: number;             // 총 토큰 예산 (비용 제어)
  timeoutMs: number;                // 전체 작업 타임아웃
  planningStrategy: "sequential" | "parallel" | "adaptive";
}

interface TaskDecomposition {
  originalTask: string;
  plan: {
    reasoning: string;              // Orchestrator의 분해 근거 (transparency)
    subtasks: SubTask[];
    dependencies: DependencyGraph;  // DAG로 실행 순서 결정
    estimatedCost: TokenEstimate;
  };
}

interface SubTask {
  id: string;
  workerType: WorkerRole;           // 어떤 Worker에게 할당할지
  instruction: string;              // Worker에게 전달할 구체적 지시
  inputArtifacts: ArtifactRef[];    // 선행 Worker의 산출물 참조
  outputSchema: JSONSchema;         // 산출물의 기대 형식 (검증용)
  priority: number;
  estimatedTokens: number;
}

B. Worker — Specialized Execution Unit

interface WorkerConfig {
  role: WorkerRole;
  model: string;                    // 역할에 맞는 모델 선택 (비용 최적화)
  systemPrompt: string;             // 역할 특화 프롬프트
  tools: ToolDefinition[];          // 역할 특화 도구 세트
  retrieval?: RetrievalConfig;      // 역할 특화 지식 베이스
  outputSchema: JSONSchema;         // 산출물 스키마 (강제)
  maxRetries: number;               // 실행 실패 시 재시도 횟수
  verificationGate?: VerificationFn; // 산출물 검증 함수
}

// Worker Role 정의 예시
type WorkerRole =
  | "researcher"      // 정보 수집 및 분석
  | "coder"           // 코드 생성
  | "reviewer"        // 코드/문서 리뷰
  | "tester"          // 테스트 생성 및 실행
  | "writer"          // 문서 작성
  | "analyst"         // 데이터 분석
  | "planner";        // 세부 계획 수립

C. Verification Layer — Hallucination 차단

interface VerificationGate {
  type: "schema" | "execution" | "llm-judge" | "human";

  // Schema Validation: 산출물이 기대 형식을 준수하는지
  schemaCheck(output: any, schema: JSONSchema): ValidationResult;

  // Execution Verification: 코드를 실제 실행하여 동작 확인
  executionCheck(code: string, testCases: TestCase[]): ExecutionResult;

  // LLM Judge: 다른 LLM이 품질을 평가
  llmJudge(output: any, criteria: string): QualityScore;

  // Human-in-the-Loop: 중요 결정에 대한 사람 승인
  humanApproval(output: any, context: string): Promise<boolean>;
}

// Retry 전략
interface RetryPolicy {
  maxAttempts: number;              // MetaGPT는 3회 사용
  strategy: "same-worker" | "escalate" | "alternative-worker";
  feedbackMode: "error-only" | "full-context";  // 실패 정보 전달 방식
}

D. Shared State Store — 구조화된 Artifact 관리

interface SharedStateStore {
  // MetaGPT의 Shared Message Pool + Pub/Sub 개념 구현

  publish(artifact: StructuredArtifact): void;
  subscribe(role: WorkerRole, filter: ArtifactFilter): Observable<StructuredArtifact>;

  // Artifact는 자연어가 아닌 구조화된 데이터
  interface StructuredArtifact {
    id: string;
    producedBy: WorkerRole;
    type: ArtifactType;             // "requirements" | "design" | "code" | "test_result"
    schema: JSONSchema;             // self-describing
    content: Record<string, any>;   // schema-validated content
    metadata: {
      timestamp: Date;
      tokenCost: number;
      version: number;
      dependsOn: string[];          // DAG edge
    };
  }
}

5.4 Worker Lambda 구현 예시

// worker-research.ts — Research Worker Lambda
import { BedrockRuntimeClient, InvokeModelCommand } from "@aws-sdk/client-bedrock-runtime";
import { DynamoDBClient, PutItemCommand } from "@aws-sdk/client-dynamodb";

const bedrock = new BedrockRuntimeClient({ region: "us-east-1" });
const dynamo = new DynamoDBClient({ region: "us-east-1" });

interface WorkerInput {
  taskId: string;
  sessionId: string;
  instruction: string;
  inputArtifacts: Array<{ key: string; content: string }>;
  outputSchema: Record<string, any>;
}

interface WorkerOutput {
  taskId: string;
  workerRole: string;
  result: Record<string, any>;
  metadata: {
    tokensUsed: number;
    latencyMs: number;
    model: string;
  };
}

export async function handler(event: WorkerInput): Promise<WorkerOutput> {
  const startTime = Date.now();

  // 1. Context Assembly — 선행 artifact를 컨텍스트에 포함
  const contextBlock = event.inputArtifacts
    .map(a => `\n${a.content}\n`)
    .join("\n");

  // 2. LLM Invocation — 역할 특화 프롬프트
  const response = await bedrock.send(new InvokeModelCommand({
    modelId: "anthropic.claude-sonnet-4-6-20250514-v1:0",
    contentType: "application/json",
    body: JSON.stringify({
      anthropic_version: "bedrock-2023-05-31",
      system: `You are a research specialist. Your output MUST conform to this JSON schema:
${JSON.stringify(event.outputSchema, null, 2)}

Rules:
- Only return factual, verifiable information
- Cite sources where possible
- If uncertain, explicitly state uncertainty level
- Never fabricate data`,
      messages: [{
        role: "user",
        content: `Context from prior steps:\n${contextBlock}\n\nTask: ${event.instruction}`
      }],
      max_tokens: 4096,
      temperature: 0.2  // Research는 낮은 temperature
    })
  }));

  const body = JSON.parse(new TextDecoder().decode(response.body));
  const resultText = body.content[0].text;

  // 3. Output Parsing & Schema Validation
  const result = JSON.parse(resultText);
  // Schema validation은 Verification Gate에서 수행

  // 4. State 기록
  await dynamo.send(new PutItemCommand({
    TableName: "orchestrator-artifacts",
    Item: {
      sessionId: { S: event.sessionId },
      taskId: { S: event.taskId },
      workerRole: { S: "researcher" },
      output: { S: JSON.stringify(result) },
      timestamp: { N: String(Date.now()) },
      ttl: { N: String(Math.floor(Date.now() / 1000) + 86400) }
    }
  }));

  return {
    taskId: event.taskId,
    workerRole: "researcher",
    result,
    metadata: {
      tokensUsed: body.usage.input_tokens + body.usage.output_tokens,
      latencyMs: Date.now() - startTime,
      model: "claude-sonnet-4-6"
    }
  };
}

6. Architecture Trade-offs & Decision Matrix

6.1 언제 Orchestrator-Worker를 사용하는가?

Use When (적용 조건):

  • 작업의 하위 구성요소를 미리 결정할 수 없을 때 (동적 분해 필요)
  • 각 하위 작업이 서로 다른 전문성/도구/지식을 요구할 때
  • 품질이 비용보다 중요할 때 (MetaGPT: +62% 토큰, -67% 인간 수정)
  • 중간 결과의 검증이 가능하고 의미있을 때
  • 단일 LLM 호출로는 컨텍스트 윈도우가 부족하거나 품질이 떨어질 때

Don't Use When (비적용 조건):

  • 단일 LLM 호출로 충분한 품질이 나올 때 (Anthropic: "start simple")
  • 작업 분해가 정적이고 예측 가능할 때 → Parallelization으로 충분
  • Latency가 최우선 제약일 때 (Orchestrator overhead 불가피)
  • 비용을 예측 가능하게 고정해야 할 때 (동적 Worker 수 = 가변 비용)
  • 시스템의 실패 모드를 완전히 열거해야 할 때 (non-deterministic)

7.2 Critical Success Factors

1. Tool Schema Quality

Anthropic이 강조: "도구 최적화에 프롬프트 튜닝보다 더 많은 시간을 투자하라." Worker가 사용하는 tool의 description, parameter schema, error handling이 전체 성능의 병목.

2. Structured Communication

MetaGPT가 증명: 자연어 → 구조화 포맷 전환만으로 executability 2.25→3.75. Worker 간 통신을 반드시 JSON Schema로 강제.

3. Verification Gates

실행 검증만으로 +4.2~5.4%p 성능 향상. 모든 Worker output에 최소 schema validation, 가능하면 execution test 적용.

4. Budget & Timeout Discipline

동적 분해의 최대 리스크는 비용 폭발. 반드시 per-request budget cap, per-worker timeout, total iteration limit 설정.

7.4 핵심 원칙 요약

From Theory to Production: 4가지 핵심 교훈

  1. Anthropic: "Start simple, add complexity only when simple fails" — Phase 1에서 baseline 측정 후 점진적 확장
  2. Andrew Ng: "Focus beats capacity" — Worker는 하나의 역할에만 집중. Multi-tool worker보다 single-purpose worker가 우수
  3. LangChain: "Supervisor-as-tool-user" — Worker를 tool로 등록하면 기존 tool-use 메커니즘 재활용 가능. 새 인프라 불필요
  4. MetaGPT: "Structured artifacts + execution verification = hallucination prevention" — 이 두 가지가 품질의 80%를 결정
