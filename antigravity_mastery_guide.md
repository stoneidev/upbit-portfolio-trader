# 🚀 Google Antigravity 마스터 가이드: AI 에이전트와 완벽히 협업하는 법

> [!NOTE]
> 이 가이드는 Google과 DeepMind가 공동 개발한 Agent-First 개발 플랫폼 **Google Antigravity**를 가장 효과적으로 활용하기 위한 실전 팁, 공식 문서 리소스, 핵심 아키텍처 및 고급 워크플로우를 정리한 아티클입니다.

---

## 1. 🌟 Google Antigravity란 무엇인가?

기존의 AI 코딩 도구(GitHub Copilot 등)가 단순한 '자동 완성(Autocomplete)'이나 대화형 챗봇에 머물렀다면, **Google Antigravity**는 한 단계 더 나아간 **'Agent-First' 개발 플랫폼**입니다.

*   **자율적 실행력:** 단순한 코드 제안을 넘어, 개발자가 지시한 목표를 달성하기 위해 **스스로 계획을 세우고(Plan), 코드를 작성 및 수정하며(Execute), 브라우저나 CLI 환경에서 정상 작동 여부를 검증(Validate)**하는 자율 순환 루프를 수행합니다.
*   **DeepMind의 강력한 추론 엔진:** 구글 DeepMind의 최신 Gemini 모델 아키텍처를 기반으로 설계되어, 수백 개의 파일로 이루어진 대규모 코드베이스에서도 정교한 의존성 분석과 논리 전개가 가능합니다.

---

## 2. 🤖 LLM vs AI Agent: 무엇이 다를까?

많은 사람들이 단순 LLM(대형 언어 모델)을 채팅창에서 사용하는 것과 Antigravity 같은 **AI Agent**를 사용하는 것의 차이를 헷갈려합니다. 두 방식은 설계 철학과 작동 방식에서 근본적인 차이가 있습니다.

| 구분 | 💬 단순 LLM 사용 (Pure LLM) | 🤖 AI 에이전트 (AI Agent) |
| :--- | :--- | :--- |
| **작동 패턴** | **단발성 응답 (One-shot)**<br>질문(Prompt)을 하면 한 번에 답을 출력함 | **자율적 순환 루프 (Sense-Plan-Act)**<br>목표를 세우고 결과에 맞춰 행동을 수정함 |
| **상태 관리 (State)** | **Stateless (상태 없음)**<br>대화 맥락이 끊기거나 기억이 한정적임 | **Stateful (상태 유지)**<br>`task.md` 등을 통해 자신의 상태와 목표를 추적 |
| **도구 활용 (Tool Use)** | **불가 또는 제한적**<br>코드 텍스트만 보여주고 사용자가 직접 적용해야 함 | **다양한 외부 도구 자율 사용**<br>파일 읽기/쓰기, CLI 명령 실행, 브라우저 테스트 등 |
| **오류 수정 (Self-Healing)** | **사용자의 몫**<br>빌드 에러가 나면 사용자가 에러 로그를 다시 입력해야 함 | **자가 교정 (Self-correction)**<br>빌드 에러 발생 시 스스로 로그를 파싱해 코드 재수정 |
| **협업 스타일** | **지시자 - 도구**<br>한 줄 한 줄 마이크로 매니징 필요 | **동료 개발자 (Collaborator)**<br>바이브 코딩(Vibe Coding) 및 설계도 승인 기반 |

### 💡 왜 에이전트가 강력한가?
단순 LLM은 **"답을 아는 챗봇"**이지만, AI 에이전트는 **"일을 실행하는 일꾼"**입니다. 에이전트는 계획을 세우고, 코드가 동작하는지 직접 터미널에 빌드 및 테스트를 돌려 확인하며, 에러가 나면 스스로 코드를 고칩니다. 이 과정에서 개발자는 복잡한 구현 과정을 하나하나 지시할 필요 없이, **설계 승인과 피드백**에만 집중할 수 있게 됩니다.

## 3. 🔄 핵심 작동 원리: 4단계 개발 루프와 아티팩트(Artifacts)

Antigravity의 가장 핵심적인 특징은 사람과 에이전트 간의 **"신뢰"**를 형성하는 **아티팩트 기반 워크플로우(Artifact-based Workflow)**입니다.

![Antigravity Workflow Diagram](./antigravity_workflow.svg)

### 📄 3대 핵심 아티팩트 (Artifacts)
에이전트는 작업을 진행하며 로컬 디렉토리(보통 `~/.gemini/antigravity/brain/<conv-id>/` 하위)에 다음과 같은 마크다운 산출물을 실시간으로 기록하고 갱신합니다.

| 아티팩트 명 | 역할 및 중요성 | 활용 팁 |
| :--- | :--- | :--- |
| **`implementation_plan.md`** | 기술 설계도 및 구현 계획서 | 본격적으로 코드를 작성하기 전에 수정할 파일 범위, 의존성 관계, 변경 로직을 정의합니다. **이 단계를 꼼꼼히 확인하고 피드백을 주는 것이 협업 성공의 70%를 결정합니다.** |
| **`task.md`** | 실시간 작업 체크리스트 | 구현해야 할 태스크들을 세부 할 일 목록으로 쪼개어 보여줍니다. `[ ]`(대기), `[/]`(진행중), `[x]`(완료) 상태를 통해 현재 에이전트가 어떤 부분을 작업하고 있는지 실시간으로 모니터링할 수 있습니다. |
| **`walkthrough.md`** | 작업 완료 및 검증 결과서 | 구현 완료 후 수정된 코드의 전체 변경 내역(Diffs)과 로컬 테스트 빌드 결과, 스크린샷 및 브라우저 테스트 비디오 녹화본 등을 제공하여 개발자가 직관적으로 결과물을 검증할 수 있도록 돕습니다. |

## 4. 🚀 Antigravity 2.0의 혁신과 핵심 기능

2026년 Google I/O에서 공식 발표된 **Antigravity 2.0**은 기존 IDE 플러그인 형태를 벗어나 독립적인 **'개발자 에이전트 OS'** 수준으로 진화했습니다.

> [!IMPORTANT]
> **Antigravity 2.0의 주요 변화 요약**
> *   **IDE 독립성:** 전용 데스크톱 클라이언트(Desktop App)로 분리되어 미션 컨트롤 대시보드인 **Agent Manager**를 제공합니다.
> *   **Antigravity CLI:** 마우스 없이 터미널 내에서 빠르고 간결하게 에이전트를 구동하고 태스크를 제약 없이 처리할 수 있습니다.
> *   **Antigravity SDK (Python):** 개발자가 자사 서비스나 연구용 환경에 맞추어 독자적인 커스텀 에이전트 파이프라인을 직접 코딩하고 배포할 수 있습니다.
> *   **Managed Agents (구글 클라우드 샌드박스):** 단 한 번의 API 호출로 안전하게 격리된 Linux 컨테이너 환경에서 코드를 실행, 테스트하고 빌드할 수 있습니다.

### 🤖 특화 서브에이전트(Subagents) 기능
에이전트는 하나의 큰 문제를 풀기 위해 필요한 경우, 스스로 특정 업무에 특화된 **서브에이전트(Subagents)**를 만들어 위임할 수 있습니다.
*   **`browser_subagent` (브라우저 서브에이전트):** 웹 UI 테스트, 최신 기술 문서 크롤링, 외부 API 가이드 분석 등을 담당합니다.
*   **`cli_subagent` (터미널 서브에이전트):** 빌드 오류 메시지 분석, 테스트 명령어 순차 실행 등을 병렬 처리합니다.

## 5. 💡 Antigravity를 200% 활용하는 실전 팁 (Best Practices)

### ① 바이브 코딩(Vibe Coding)을 위한 기획 단계 피드백 극대화
*   에이전트가 코드를 다 짜고 나서 수정하려면 리소스 낭비가 큽니다.
*   **Tip:** `implementation_plan.md`가 생성되었을 때 승인(Approve)하기 전, 수정하고 싶은 파일 경로, 적용할 라이브러리 버전, 코드 스타일 아키텍처(예: Domain-Driven Design, Layered Architecture 등)에 대해 **자세히 피드백을 주어 계획서를 먼저 정교하게 튜닝**하세요.

### ② `AGENTS.md` & `GEMINI.md` 설정 최적화
*   프로젝트의 루트 디렉토리에 정책 파일을 만들어 에이전트가 항상 지켜야 할 규칙을 명시합니다.
    *   `AGENTS.md`: "이 프로젝트는 Java 17 및 Spring Boot 3.x를 사용합니다. 모든 서비스 계층은 DDD 원칙을 따르고, DTO 변환은 MapStruct를 사용합니다."와 같은 **스타일 가이드라인** 설정.
    *   `GEMINI.md`: 보안상 실행을 제한할 터미널 명령어나 폴더 경로를 지정하는 **Allow/Deny 정책** 설정.

### ③ 브라우저 서브에이전트 적극 유도
*   프론트엔드 작업이나 UI 디자인 수정 시, 에이전트에게 **"실제 브라우저를 열어서 UI의 레이아웃이나 반응형 디자인이 깨지지 않는지 browser_subagent로 확인하고 결과를 알려줘"**라고 요청해보세요.
*   에이전트가 직접 웹 브라우저를 컨트롤하여 검증하고, 스크린샷 및 비디오 링크를 포함한 결과물을 제공해 줍니다.

### ④ 보안 및 Allow List 환경 관리
*   로컬 머신에서 작동하므로 에이전트가 destructive(파괴적인) 명령어나 스크립트를 실수로 실행하지 않도록 주의해야 합니다.
*   **Tip:** 위험 가능성이 있는 명령어(예: 데이터베이스 스키마 드롭, 외부 패키지 글로벌 설치 등)는 에이전트가 실행 전 반드시 **사용자 승인 요청(Ask for approval)**을 하도록 설정되어 있습니다. 터미널 출력을 늘 꼼꼼하게 읽는 습관을 들여야 합니다.
